基于本地大模型的 Agent 链式工具调用系统

先说说为什么要做这个东西。现在市面上很多 AI 编程助手或者 Agent 框架，用起来都有些让人头疼的地方。比如数据隐私风险大，因为它们太依赖在线 API 了，你得把自己的代码、文档这些敏感东西传到第三方服务器上去。再比如它们不支持本地大模型，大部分都是针对云服务设计的，你没法自己换个本地的模型来用。还有就是工具调用不够灵活，很多系统只能一次调一个工具，不能自动规划好几步的任务。最后就是学习成本太高了，像 LangChain 那种配置特别复杂，初学者看着就头大。

从测试结果看，我做的这个系统算是把上面这些问题都解决了一些。实际跑起来有几个特点。第一个是本地部署保住了数据安全，我用了 LMStudio 来本地跑大模型，比如 qwen3-coder，不需要联网就能用核心功能。第二个是设计了一个链式工具调用的框架，弄了个叫 ChainedCallContext 的上下文管理器，可以实现多步骤任务的自动规划和执行。第三个是集成了好几种函数调用，包括文件操作、访问网页、查本地知识库等等。第四个是用 SPEC 范式来规范开发流程，这样写起来不容易乱。第五个是支持流式输出，可以实时看到大模型生成的内容。

这个系统是用 Python 写的，基于 OpenAI 兼容的 API 来调用本地或者在线模型，主要实现了链式调用循环、JSON 解析和验证、工具的注册和执行，还有一个可以交互的命令行界面。

再说说文献和对比。我查了一些资料，发现链式工具调用在 AI Agent 里确实有不少好处。比如有人写了篇 AutoGPT 的论文，讲怎么让大模型自己规划任务；还有一篇中文综述专门讲工具学习；OpenAI 自己也有一篇介绍函数调用的文章。市面上比较常见的有 ChatGPT 纯对话、LangChain 和 AutoGPT。ChatGPT 虽然好用，但不支持本地模型，也不能链式调用，数据安全性一般。LangChain 虽然能本地化也能链式调用，但用起来太复杂了。AutoGPT 能链式调用，但也是依赖在线服务。我这个系统在易用性上还不错，同时支持本地化和链式调用，数据安全性也比较好。

接下来讲讲项目是怎么做的。整个项目的文件结构是这样的：practice01 文件夹里有 llm_test.py、web_summarizer.py 以及 requirement.md、spec.md、api.md 这些文档。practice02 里有 chat_stream.py 和 chat_tools.py。practice03 里有 chat_search.py 和 chat_summary.py。practice04 里主要就是 chat_summary.py，集成了 AnythingLLM。practice05 里有 test_notice_skill.py 和 chat_summary.py，用来测试技能加载。practice06 里就是完整的链式工具调用实现 tool_client.py。

核心的技术点有几个。一是模型部署，用 LMStudio 本地跑 qwen2.5-7B-Instruct、qwen3-coder 这些模型，还可以通过 OpenRouter 切换到在线模型做对比。二是本地知识库，用了 AnythingLLM，通过 curl 调用它的 API 来查本地文档。三是函数调用，包括文件的读写、列出目录、抓取网页、查询知识库。四是链式调用的上下文管理器 ChainedCallContext，它可以记录每一步的调用和结果，存中间变量，还能设置最大迭代次数防止无限循环。五是链式调用的执行器 execute_chained_tool_call，它会循环让大模型做决策，然后执行对应的工具，同时支持 JSON 和原生 tool_calls 两种格式。六是技能加载系统，会读取 .agents/skills 目录下的 SKILL.md 文件，动态地把提示词塞给大模型，比如 notice 技能和 init-spec 技能。七是 SPEC 范式，用 init-spec 技能生成 requirement.md、spec.md、api.md、test.md 这四个文档，这样开发的时候就有规范可循。八是流式输出，可以让大模型一边生成一边显示在屏幕上。

因为我是一个人做的，所有的代码编写、测试和文档都是我自己完成的。

然后是测试部分。测试的对象就是链式工具调用系统，主要是 practice06 里的功能。测试环境是 Windows 10，Python 3.12，LMStudio 本地加载 Qwen2.5-7B-Instruct 模型，电脑内存 16GB。我设计了五个测试用例。第一个是文件搜索链式调用，让它查找包含 def 的 Python 文件并总结。第二个是多文件数字求和，让它读 1.txt 和 2.txt 里的数字，加在一起写到 result.txt 里。第三个是网页处理链式调用，抓取一个网页的内容并生成摘要保存。第四个是技能调用，让它用 notice 技能写个通知。第五个是 SPEC 文档生成，调用 init-spec 技能生成四个文档。

测试的时候我关注了这几个指标：链式调用的成功率、平均迭代次数、平均响应时间、工具调用的准确率。实际跑下来，文件搜索的成功率是百分之百，平均迭代了三点二次，平均响应时间四点三秒，工具调用准确率也是百分之百。数字求和的成功率百分之百，平均迭代三点零次，响应时间三点八秒，准确率百分之百。网页处理的成功率百分之百，平均迭代四点一次，响应时间六点二秒，准确率百分之百。技能调用的成功率百分之百，平均迭代一次，响应时间二点一秒，准确率百分之百。SPEC 生成的也是百分之百成功，平均迭代五次，响应时间八点五秒，准确率百分之百。

从这些数据来看，系统还是挺稳定的，没有出现调用失败的情况，工具的参数也都提取正确。网页处理花的时间最长，SPEC 生成迭代的次数最多，这也符合预期，毕竟网页抓取和生成多个文档的任务更复杂一些。

最后是结论。写代码的时候发现，这个系统确实能做到本地大模型驱动链式工具调用。实际用起来有这几个好的地方：本地部署保护了数据隐私，不用联网也能用；链式调用框架让大模型自己规划多步任务，自动化水平提高了；函数调用机制很灵活，可以扩展很多工具；SPEC 范式让开发维护不那么累。本地大模型完全能胜任这种复杂的工具链式调用任务，响应速度和准确率对于教学和轻量级开发来说足够了。基于上下文管理器的链式调用设计可以避免无限循环，还能在步骤之间传递变量，有点像推理-行动的那种智能体行为。集成了 AnythingLLM 知识库之后，大模型对项目文档的检索能力明显更强了，回答也更准。

当然也有不足，比如对硬件要求比较高，建议至少 16GB 内存，而且现在还只支持文本类的工具。以后可以加入视觉模型和多模态工具，还可以优化链式调用的效率，比如做缓存或者并行执行。

总的来说，这个系统针对当前 AI Agent 框架太依赖在线 API、隐私风险高、工具调用不够灵活这些问题，搞了一个基于本地大模型的链式工具调用方案。用 Python 开发，通过 LMStudio 本地跑大模型保障数据安全，设计了 ChainedCallContext 来管理多步任务，集成了文件、网页、知识库等工具，还用 SPEC 范式规范了开发。从测试结果看，所有用例都成功了，工具调用也都很准，系统挺可靠的。实际用下来，本地大模型在复杂链式工具调用任务中是可行的，给轻量级、隐私安全的 AI 助手开发提供了一个蛮不错的思路。
